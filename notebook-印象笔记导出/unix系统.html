CSS base grammar:(层叠样式表单，它是一种用来表现HTML或XML等文件格式的program lan)
  • CSS定义：selector,properties,value|selector { property: value} eg: body { color: black} or p { font-family: "sans serif"} or p { text-align: center; color: red}
  • Selector 可以有多种形式（一般是定义样式的HTML标记,含边界）. eg:BODY,P,TABLE etc. 属性和值,以冒号隔开.
    • Selector teams:把相同属性和值的选择符组合起来书写，并用逗号将选择符分开 . eg: h1, h2, h3, h4, h5, h6 { color: green }
    • Class selector:把相同的类别/功能定义不同的样式;自定义类选择符时，在其名称前加点号. eg: .center { text-align: center} + <h1 class="center">

    • 不同的选择符定义相同的元素时，要考虑到不同的选择符之间的优先级。ID选择符，类选择符和HTML标记选择符，由高至低

三、find命令说明：                     
1、find命令的一般形式为；
find pathname -options [-print -exec -ok ...]

2、find命令的参数；
pathname: find命令所查找的目录路径。例如用.来表示当前目录，用/来表示系统根目录。
-print：  find命令将匹配的文件输出到标准输出。
-exec：   find命令对匹配的文件执行该参数所给出的shell命令。相应命令的形式为'command' {  } ";，注意{   }和"；之间的空格。
-ok：      和-exec的作用相同，只不过以一种更为安全的模式来执行该参数所给出的shell命令，在执行每一个命令之前，都会给出提示，让用户来确定是否执行。

3、find命令选项

-name

按照文件名查找文件。

-perm
按照文件权限来查找文件。

-prune
使用这一选项可以使find命令不在当前指定的目录中查找，如果同时使用-depth选项，那么-prune将被find命令忽略。

-user
按照文件属主来查找文件。

-group
按照文件所属的组来查找文件。

-mtime -n +n
按照文件的更改时间来查找文件， - n表示文件更改时间距现在n天以内，+ n表示文件更改时间距现在n天以前。find命令还有-atime和-ctime 选项，但它们都和-m time选项。

-nogroup
查找无有效所属组的文件，即该文件所属的组在/etc/groups中不存在。

-nouser
查找无有效属主的文件，即该文件的属主在/etc/passwd中不存在。

-newer file1 ! file2
查找更改时间比文件file1新但比文件file2旧的文件。

-type
查找某一类型的文件，诸如：
b - 块设备文件。
d - 目录。
c - 字符设备文件。
p - 管道文件。
l - 符号链接文件。
f - 普通文件。

-size n：[c] 查找文件长度为n块的文件，带有c时表示文件长度以字节计。
-depth：     在查找文件时，首先查找当前目录中的文件，然后再在其子目录中查找。
-fstype：    查找位于某一类型文件系统中的文件，这些文件系统类型通常可以在配置文件/etc/fstab中找到，该配置文件中包含了本系统中有关文件系统的信息。

-mount：在查找文件时不跨越文件系统mount点。
-follow：如果find命令遇到符号链接文件，就跟踪至链接所指向的文件。
-cpio：对匹配的文件使用cpio命令，将这些文件备份到磁带设备中。

注：对于任何类型的变量，除非我们显式的初始化过，否则我们都不应该假设它拥有一个有效的地址；同样，指针也是如此。
指向无效地址（存储空间为0）的指针称为（悬）空指针；产生空指针的错误有：将任意的整型变量强制转化为指针变量，操作超出数组边界的指针以及释放一个或多个仍被引用的指针。
存储空间分配：在内存中预留存储空间的过程
  • 指针和内存分配是密不可分的。当通过指针访问内存时，其意义就如同点菜一样：指针对应菜名，内存地址作为中间变量可以忽略不计，而内存空间存储的数据对应实际的          菜。
  • 声明一个指针时，仅仅是为指针本身分配了空间，并没有为指针所引用的数据分配空间。而为数据分配空间有两种方式：一种是直接声明一个变量(初始化)；二是在运行时动态的分配存储空间。（例如：malloc,realloc）
    • 初始化：根据变量的类型预留足够的内存空间；这种存储空间是系统自动分配，但是其生命周期随作用域而变——这一点在处理自动变量是尤为重要。
    • 动态分配内存：我们会得到一个指向堆存储空间的指针。此存储空间由我们自行管理，并且会一直存在，直到我们显式释放。

注：有些时候动态分配内存会带来一些危险：尤其是动态内存分配造成了内存泄露问题。内存泄露的产生是由于动态分配了内存空间，但从未释放它。特别在重复代码时，这种问题尤为严重。
      采用统一的内存管理方式可以大大减少此类问题。

内存管理方法——采用统一的数据结构
由用户来管理存储空间【注：分静态/动态分配】（用于相应的数据结构和变量），而数据结构本身只维护内部变量【注：指向用户分配存储空间的指针】的存储空间分配。所以在数据结构中，只使用指针来指向数据变量，而不使用此数据的私有副本。
采用统一的初始化/销毁数据结构的操作。初始化可能会涉及很多的步骤，其中之一就是内存分配；销毁数据结构通常包括：删除它的所有数据变量（用户分配）以及释放数据结构本身所用到的内存。（当然，用户也可以自己管理数据的内存，而不是野蛮删除）。
每个数据结构在初始化的时候，都需要使用用户提供的初始化函数。但是，在释放内存时可以有灵活的选择。——数据存储的管理实际上是一种与具体应用相关的操作，需要一定的跨平台性。
一个数据结构的实现并不依赖于它所存储数据的类型和大小；同时，多个数据结构能够以单个数据形态表示，这个特性在组织大量数据时非常有用。

为什么使用vagrant？

vagrant尝试替代的工作方式：在vagrant出现之前，开发一个WEB应用需要安装和配置所有所需的软件。当然，是在你的本地机器上。

不过“本地机器”会给你带来数不尽的问题。
  • 所有的东西都必须手动安装。针对WINDOWS AND MAC OSX平台：安装某些软件可行，但是大量的软件都不能编译和运行（仅UNIX）。
  • 配置比安装更为复杂。一旦软件安装完毕，它需要配置。但困难在于开发者不可能知道如何配置软件以使之正常工作在生产环境。错误配置的软件在开发环境和生产环境的行为不同。
  • 手工搭建的开发环境可能不是生产环境的真实写照。否则，软件可能在开发环境中正常工作，但是在生产环境中出现BUGS。
  • 在一台机器上开发多个项目几乎不可能。因为每一个项目都需要相当不同的配置，或是需要完全不同的后端系统。但仍要如此，最后的结果就是：系统所配置的依赖缺乏健壮性；系统运行大量的服务以至于每一个独立的应用不完全需要。
  • 保持自己的开发环境与其他成员同步是相当困难的。若每一个开发者都可以隔离开发环境，那同步是一件容易的事。而这可以加速项目的开发。
  • 在同一项目中，多个开发者使用不同的OS是相当困难的。一些软件在某个OS上几乎不可能运行。因此，开发者通常会以有良好文档的OS作为开发环境。

针对上述问题，vagrant是一个现代的解决方案。在大多数情况下，vagrant提倡自动化。在搭建开发环境时，使用Shell脚本或是配置管理工具（Chef 以及 Puppet ）。

vagrant 将你的开发环境放入一个虚拟机中，使同时开发多个项目是容易的。因为每一个项目都可以是一个虚拟机。

  6. 最为精华的部分是：1~5可以移植到每一个项目中。不论是项目1、项目2；还是工程A、工程B。只要他们遵守vagrant之道，那么工作流就是相同的。结果，产品级的遥远无期以及只能在自己的机器上工作的BUG不会出现。vagrant就像是模子，可以被模制成你所适合的开发环境。
vagrant替代方案

其他的技术也可以实现vagrant的功能。不过，这些技术都只是实现vagrant的一部分功能。而这带来的最大弊端就是丧失vagrant所强制的工作流；你不得不创造自定义的工作流或是接近于vagrant的工作流。
目前，vagrant可以工作在任意的虚拟层中；这些虚拟软件（例如VMwarre）提供了官方的选项支持。vagrant甚至可以在远程环境中运行（例如EC2）。

  • 常规的桌面虚拟环境
          vagrant可以视为虚拟解决方案的顶层，所以你可以直接使用这些虚拟解决方案。VirtualBox是免费的，还有一些付费版本（VMwarre或Parallels）。每一个解决方案都很接近于vagrant的功能。
          这些工具与vagrant相比主要的缺点就是： vagrant使实现的这些功能的方法变得规范化；跨平台的支持也很重要。除此之外，你也会丧失 vagrant中规中矩的工作流。
  • 容器
          另一个方法就是使用基于容器的解决方案（LXC或OpenVZ）。容器并不会提供完整的虚拟环境，相反它会在运行的内核上创建一个封闭的空间（也就是说容器和其他进程一样，共享同一内核）。
          容器所带来的巨大优势是：它们的开销较虚拟环境来说小得多。
          当然容器的劣势也是十分明显的：你只能在容器中运行“与宿主机相同的内核”。所以，你不可能在Linux宿主机中运行windows容器。可见在开发windows和OSX应用程序上，容器的局限相当大。
          容器在开发过程中的另一个劣势就是：它对宿主OS十分挑剔。 linux支持LXC或OpenVZ，FreeBSD支持Jails。OSX与windows都不提供容器（Docker服务器）支持。因此容器的使用需要团队使用同一个OS，最好是linux。
          总体上来说，容器对于产品部署上有一定的帮助（因为容器提供了一个安全策略来封装同一OS的资源，性能开销也相对较小）。但是在开发过程中，容器的严格限制使之很难完美。   
  • 云
          最后一个方法就是抛弃桌面虚拟环境，直接在云上搭建开发环境。
          这种方法主要的优势在于它允许开发者在令人难以想象的极低配置的桌面上工作。当然云环境要（极接近于产品级） 。有些时候，在要求产品环境相对高级的情况下，云是一种不错的选择。
          缺点是时刻需要网络连接，付费昂贵。不过对于 vagrant来说，云和虚拟环境是“等同”的。无论是工作流，还是配置。

你可以简单的以下列指令开始。（模式：vagrant + 子命令）。而这些命令的参数都会配置vagrantfile的某些参数。

vagrantfile

每个项目都有自己的隔离的工作环境，因而vagrant针对每个项目都会被配置（对应vagrantfile）。
vagrantfile是一个简单的TEXT文件以决定创建工作环境的所需。这个文件描述了你想要引导的OS类型、虚拟机的物理特性（例如RAM）、虚拟机上的软件安装以及多种访问虚拟机的网络配置。

vagrantfile应该置于版本控制之下。这是一种重构的方式（你的团队成员可以以此快速创建工作环境，而无需任何配置）。经验表明，vagrantfile的设置需要一点点的精力，但是你的团队成员却受益匪浅。

vagrantfile是通用的，这意味着它们可以使用在vagrant所支持的各个平台。其更加简化了创建开发环境的工作，因为一旦vagrantfile已经写好，就可以应用在各个OS。

vagrantfile 通常不用修改。因为与正在开发的软件的实际代码库相比，开发环境的底层设置是相对稳定的。所以，只有当OS镜像发生改变时，你需要修改vagrantfile；除此之外，创建开发环境就是一个把vagrantfile迭代的过程，
之后vagrant会快速节省你所花费的时间（实现构建稳定、复杂开发环境的自动化，详细参见vagrant之道）。

定义

嵌入式系统是硬件和软件的结合，或许同时还包含一些其他的附加部分，设计出的系统一般是机械的或是电器化的（控制系统），这些系统用来实现某种特定的功能。

通常，嵌入式系统是大型系统内部的组件；在大多数情况下，多个嵌入式系统是由一个通信网络所连接的。

需要指出的是通用计算机包括多个嵌入式系统，比如键盘和鼠标。而外围设备都是由一个处理器和软件来实现一个具体的功能（换句话说外围设备的具体功能是可以用一句话来总结的）。

我们是可以设计出一个不包含处理器和软件的功能设备的。比如可以用一个定制的集成电路（IC）来代替处理器和软件的组合。二者之间的差距就是‘弹性’。

静态链接库与动态链接库的区别

 

库

  • 动态链接库：LIB包含 函数所在的DLL文件和DLL文件中函数的位置信息，而DLL则包含函数代码本身（在运行时加载在进程空间）。
  • 静态链接库：LIB包含函数代码本身，在编译时直接将代码加入程序中。

 

链接方式

  • 动态链接：使用动态链接库，允许可执行文件（.dll或.exe）仅包含 在运行时定位DLL函数所需的位置信息。
  • 静态链接：使用静态链接库，链接器从静态链接库拷贝所有被调用函数，并将库函数同代码一起放到可执行文件。

 

lib和dll的区别：

  • lib是编译时用到的，dll是运行时用到的。如果要完成源代码的编译，只需要lib；如果要使动态链接的程序运行起来，只需要dll。
  • 如果有dll文件，那么lib一般是一些索引信息，记录了dll中函数的入口和位置，dll中是函数的具体内容；如果只有lib文件，那么这个lib文件是静态编译出来的，索引和实现都在其中。（使用静态编译的lib文件，在运行程序时不需要再挂动态库，缺点是导致应用程序比较大，而且失去了动态库的灵活性）。
  • 动态链接的情况下，有两个文件：一个是LIB文件，一个是DLL文件。LIB包含被DLL导出的函数名称和位置，DLL包含实际的函数和数据，应用程序使用LIB文件链接到DLL文件。在应用程序的可执行文件中，存放的不是被调用的函数代码，而是DLL中相应函数代码的地址，从而节省了内存资源。DLL和LIB文件必须随应用程序一起发行，否则应用程序会产生错误。

 

使用lib需注意两个文件：

  • .h头文件，包含lib中说明输出的类或符号原型或数据结构。应用程序调用lib时，需要将该文件包含入应用程序的源文件中。
  • .LIB文件，实现代码。

 

使用dll需注意三个文件：

  • .h头文件，包含dll中说明输出的类或符号原型或数据结构的.h文件。应用程序调用dll时，需要将该文件包含入应用程序的源文件中。
  • .LIB文件，是dll在编译、链接成功之后生成的文件，作用是当其他应用程序调用dll时，需要将该文件引入应用程序，否则产生错误。
  • .dll文件，真正的可执行文件，开发成功后的应用程序在发布时，只需要有.exe文件和.dll文件，并不需要.lib文件和.h头文件。
